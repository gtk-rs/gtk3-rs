fn connect_changed<F: Fn(&Self, &File, Option<&File>, FileMonitorEvent) + 'static>(
    &self,
    f: F
) -> SignalHandlerId[src]

Expand description

Emitted when file has been changed.

If using [crate::FileMonitorFlags::WatchMoves] on a directory monitor, and the information is available (and if supported by the backend), event_type may be crate::FileMonitorEvent::Renamed, crate::FileMonitorEvent::MovedIn or crate::FileMonitorEvent::MovedOut.

In all cases file will be a child of the monitored directory. For renames, file will be the old name and other_file is the new name. For “moved in” events, file is the name of the file that appeared and other_file is the old name that it was moved from (in another directory). For “moved out” events, file is the name of the file that used to be in this directory and other_file is the name of the file at its new location.

It makes sense to treat crate::FileMonitorEvent::MovedIn as equivalent to crate::FileMonitorEvent::Created and crate::FileMonitorEvent::MovedOut as equivalent to crate::FileMonitorEvent::Deleted, with extra information. crate::FileMonitorEvent::Renamed is equivalent to a delete/create pair. This is exactly how the events will be reported in the case that the [crate::FileMonitorFlags::WatchMoves] flag is not in use.

If using the deprecated flag [crate::FileMonitorFlags::SendMoved] flag and event_type is crate::FileMonitorEvent::Moved, file will be set to a crate::File containing the old path, and other_file will be set to a crate::File containing the new path.

In all the other cases, other_file will be set to [crate::None].

file

a crate::File.

other_file

a crate::File or [crate::None].

event_type

a crate::FileMonitorEvent.
